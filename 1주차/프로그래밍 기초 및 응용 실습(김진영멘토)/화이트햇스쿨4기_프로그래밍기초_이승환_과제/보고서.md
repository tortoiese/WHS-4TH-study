## 1. AST란 무엇인가

**AST(Abstract Syntax Tree, 추상 구문 트리)** 는 소스코드를 컴파일러가 이해할 수 있도록
**문법 구조를 트리(나무) 형태로 표현**한 것이다.

예를 들어 `int main() { return main1(); }` 라는 코드는 다음과 같은 트리로 표현된다.

```
FuncDef (함수 정의)
├── decl: Decl (name = "main")
│   └── type: FuncDecl
│       └── type: TypeDecl → IdentifierType(["int"])   // 리턴타입 int
└── body: Compound (함수 본문 { })
    └── Return
        └── FuncCall (main1())
```

이 트리를 JSON으로 직렬화한 것이 바로 `ast.json` 이다.
각 노드는 `"_nodetype"` 키로 종류(FuncDef, Decl, If, Return ...)를 구분하고,
자식 노드를 키-값 / 배열로 가지고 있다. 즉 **소스코드를 "데이터"처럼 다룰 수 있게** 된 것이다.

이렇게 코드를 데이터로 다루면, 코드를 직접 실행하지 않고도
"함수가 몇 개인지", "어떤 타입을 쓰는지", "버그 패턴이 있는지" 등을 **자동으로 분석**할 수 있다.
(실제로 정적 분석 도구, 컴파일러, 보안 취약점 탐지기가 모두 AST를 사용한다.)

## 2. 분석 방법 (ast.c 설명)

멘토가 제공한 JSON 파서 라이브러리 `json_c.c` 를 사용해 `ast.json` 을 파싱하고,
트리를 따라 내려가며 필요한 정보를 추출했다.

핵심 아이디어는 다음과 같다.

| 추출 항목 | AST에서 찾는 위치 |
|-----------|-------------------|
| 함수 개수 | 최상위 `ext` 배열에서 `_nodetype == "FuncDef"` 인 노드 수 |
| 함수 이름 | `FuncDef → decl → name` |
| 리턴 타입 | `FuncDef → decl → type(FuncDecl) → type` 를 IdentifierType까지 따라감 |
| 파라미터  | `FuncDecl → args(ParamList) → params[]` 의 각 `type` 과 `name` |
| if 개수   | 함수 `body` 아래를 재귀로 훑어 `_nodetype == "If"` 인 노드 수 |

직접 만든 핵심 함수:

- `read_file()` : `ast.json` 파일 전체를 문자열로 읽음 (라이브러리의 `json_read`는 미구현이라 직접 작성)
- `obj_get()` / `arr_get()` : 객체·배열에서 값을 조용히 꺼내는 헬퍼
- `type_to_str()` : `PtrDecl→TypeDecl→IdentifierType` 처럼 겹겹이 싸인 타입을 `"char *"` 같은 문자열로 변환
- `count_if()` : 트리를 **재귀**로 내려가며 If 노드를 세는 함수

**[스크린샷 1]** — `gcc ast.c -o ast.exe` 컴파일 + `./ast.exe` 실행 화면을 여기에 첨부

## 4. 심화 — 원본 소스코드 복원

AST에는 원본 코드의 모든 구조 정보가 들어 있으므로, 트리를 거꾸로 따라가면
원본 소스코드를 거의 그대로 복원할 수 있다. (전체 결과는 `restored_source.c`, 약 737줄)

아래는 복원한 함수 **3개** 예시이다.

### (1) main / my_realloc

```c
int main()
{
  return main1();
}

char *my_realloc(char *old, int oldlen, int newlen)
{
  char *new = malloc(newlen);
  int i = 0;
  while (i <= (oldlen - 1))
  {
    new[i] = old[i];
    i = i + 1;
  }
  return new;
}
```

- `main` : 리턴타입 `int`, 파라미터 없음, if 0개 → 표의 1번과 일치.
- `my_realloc` : 리턴타입 `char *`, 파라미터 3개(`char* old, int oldlen, int newlen`) → 표의 2번과 일치.
  새 메모리를 할당하고 옛 내용을 한 바이트씩 복사하는, 직접 만든 realloc이다.

### (2) error

```c
void error()
{
  exit(1);
}
```

- 리턴타입 `void`, 파라미터 없음, if 0개 → 표의 3번과 일치. 에러 시 프로그램을 종료한다.

### (3) takechar

```c
void takechar()
{
  if (token_size <= (i + 1))
  {
    int x = (i + 10) << 1;
    token = my_realloc(token, token_size, x);
    token_size = x;
  }
  token[i] = nextc;
  i = i + 1;
  nextc = getchar();
}
```

- 리턴타입 `void`, if **1개** → 표의 4번과 정확히 일치.
  버퍼가 가득 차면(`if`) 크기를 늘린 뒤 현재 문자를 토큰 버퍼에 담는 함수이다.

**[스크린샷 2]** — 복원된 `restored_source.c` 를 연 화면을 여기에 첨부

> 복원 방법: AST의 각 노드 종류(FuncDef, If, While, BinaryOp ...)를 다시 C 문법으로
> 바꿔 출력하면 된다. 본 보고서에서는 pycparser의 역변환(c_generator)으로 검증했고,
> if 개수·파라미터가 ast.c 분석 결과와 모두 일치함을 확인했다.

## 5. 배운 점 / 어려웠던 점

- **배운 점**: 코드를 텍스트가 아니라 트리(데이터)로 보면, 사람이 일일이 읽지 않아도
  함수 개수·타입·조건문 같은 정보를 자동으로 뽑아낼 수 있다. 이것이 정적 분석/보안 도구의 기본 원리다.
- **어려웠던 점**:
  1. 타입이 `PtrDecl → TypeDecl → IdentifierType` 처럼 여러 겹으로 싸여 있어, 안쪽까지
     **재귀로 따라 들어가야** `char *` 같은 최종 타입을 얻을 수 있었다.
  2. if 개수는 본문 어디에 중첩돼 있을지 몰라, 트리 전체를 **재귀 탐색**해서 세야 했다.
  3. (환경) 소스/터미널 인코딩(UTF-8 ↔ CP949) 불일치로 한글이 깨져, 전부 UTF-8로 통일했다.

### 부록 — 챌린지(AST로 버그 패턴 탐지) 아이디어

AST를 순회하면서 특정 "위험 패턴"을 가진 노드를 찾으면 버그/취약점을 자동 탐지할 수 있다. 예:

- **`if` 안의 대입(`=`) 오용** : `If → cond` 가 `Assignment` 노드면 `==` 를 `=` 로 잘못 쓴 의심.
- **메모리 NULL 검사 누락** : `malloc` 호출 결과를 받은 변수에 대해 곧바로 `if(p==NULL)`
  검사가 없는 경우.
- **`gets`, `strcpy` 등 위험 함수 호출** : `FuncCall → name` 이 위험 함수 목록에 있으면 경고.

`ast.c` 의 `count_if()` 처럼 트리를 재귀 순회하되, 조건을 "If 노드"에서
"위험 패턴 노드"로 바꾸면 그대로 간단한 취약점 스캐너가 된다.
